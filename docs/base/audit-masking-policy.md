# 감사 로그 / 마스킹 정책

v2 `docs/audit-log-policy.md`를 v3 권한 모델 기준으로 이관한 정책이다.

개인정보 포함 화면을 생성하기 전에 반드시 이 문서를 읽는다.

## 현재 구현 상태

구현 완료 (2026-06-11). 상세는 `docs/base/common-utils-implementation.md`를 따른다.

- `@PrivacyLog(action = "...")` — 메서드에 붙이면 AOP가 `SMS.TB_PRIVACY_AUDIT_LOG`에 자동 기록
- `MaskingUtil` — maskName/maskPhone/maskRrn/maskCard + 텍스트 내 개인정보 마스킹
- `ExcelUtil.downloadExcel(...)` — 대용량 엑셀 다운로드
- 테이블 DDL: `db/oracle/03_privacy_audit_log_schema.sql` (DBA 적용 필요)

## 기본 원칙

- 개인정보, 민감정보, 권한, 대량 데이터에 접근하는 기능은 감사 로그 대상이다.
- 화면 버튼을 숨기는 것과 별개로 서버 메서드 기준으로 기록한다.
- 조회성 기능도 개인정보가 포함되면 기록한다.
- 엑셀/파일 다운로드는 목록 조회보다 더 강한 감사 대상이다.
- 감사 로그의 행위자는 `(EMP_ID, DEP_ID)`로 기록한다. `EMP_ID` 단독 기록을 금지한다.
- 실패한 요청도 가능한 범위에서 원인 추적이 가능해야 한다.

## 필수 적용 대상

| 구분 | 대상 | 예시 |
|---|---|---|
| 개인정보 목록 조회 | 이름, 전화번호, 주민번호, 고객번호가 포함된 목록 | 발송이력조회, 고객별 조회, 주민번호 조회 |
| 개인정보 상세 조회 | 목록 행 클릭 후 상세 팝업/상세 화면 | 발송 상세, 사용자 상세 |
| 비마스킹 조회 | `CAN_MASK_VIEW` 권한으로 원문을 조회하는 기능 | `/unmask` 계열 |
| 대량 반출 | 엑셀/CSV/파일 다운로드 | `/excel`, `/download`, `/export` |
| 파일 처리 | 개인정보 포함 파일 업로드/다운로드 | 발송 대상 파일 |
| 권한 변경 | 역할, 메뉴 권한 변경 | `TB_EMP_ROLE`, `TB_MENU_AUTH` 변경 |
| 사용자 정보 변경 | 사용자 등록/수정/삭제, 부서 변경 | 사용자관리, 부서관리 |
| 승인/반려 | 결재, 승인, 반려, 상태 변경 | 캠페인 승인, 발송대상 승인 |
| 발송/예약 | SMS/LMS/알림톡 발송 요청, 예약, 취소 | 발송 등록, 재발송, 발송 취소 |
| 보안 이벤트 | 로그인 성공/실패, 잠금, 비밀번호 변경 | LDAP 로그인, 계정 잠금 |

## 권한과의 연계

- 비마스킹 원문 조회는 `TB_MENU_AUTH.CAN_MASK_VIEW = 'Y'`인 역할만 가능하다.
- `CAN_MASK_VIEW` 사용 시점에는 항상 감사 로그를 남긴다.
- `CAN_DOWNLOAD` 사용(엑셀/파일 반출)도 항상 감사 로그를 남긴다.

## 액션명 작성 규칙

- 사용자가 이해할 수 있는 업무명으로 작성한다. 메뉴명 + 행위 형태를 권장한다.
- 예: `발송이력 목록 조회`, `주민번호 조회`, `발송이력 엑셀 다운로드`, `메뉴 권한 변경`, `캠페인 승인 처리`

## 마스킹 기준

- 화면 표시 데이터는 마스킹된 값만 내려준다. 화면(JS)에서 마스킹하지 않는다.
- 감사 로그에는 개인정보 원문 전체를 남기지 않는다.
- 대상 식별이 필요하면 PK, 업무번호, 마스킹된 전화번호, 마스킹된 이름처럼 최소 정보만 남긴다.
- 주민번호, 메시지 본문 전문, 대량 수신자 목록 원문은 로그에 직접 저장하지 않는다.

## 화면 생성 시 판단 기준

화면을 생성할 때 다음 세 가지를 확인하고, 하나라도 Y이면 해당 메서드에 감사 로그를 적용한다.

```text
개인정보 포함 여부: Y/N
엑셀 다운로드 포함 여부: Y/N
권한/승인 기능 포함 여부: Y/N
```

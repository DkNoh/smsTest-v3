# 도메인 경계 판별 가이드

화면을 만들기 전에 "무엇을 도메인 1개로 묶을지"를 정하는 기준이다.
누가 판단해도, 어떤 모델이 판단해도 같은 결과가 나오게 하는 것이 목적이다.

이 문서는 **판별 방법론**이다. 도메인별 테이블 지식은 `domain-rules.md`를 본다.

## 왜 필요한가

두 가지 실패를 동시에 막기 위해서다.

- **과결합(god service)**: 관련 없는 업무를 한 Service/Mapper에 다 때려박는다. 파일은 적지만 유지보수 불가.
- **과분해(파일 폭발)**: 거래마다, 참조 테이블마다 클래스를 만든다. 화면 30개가 수백~수천 파일이 된다.

도메인을 올바른 입자로 나누면 둘 다 피한다.

## 핵심 한 줄

> **도메인 = 그 기능이 쓰기(INSERT/UPDATE/DELETE)로 소유하는 메인 테이블(군) 1개.**
> **거래는 파일이 아니라 메서드다.**

## 용어 (혼동 금지)

| 용어 | 정체 | 기준 축 |
|---|---|---|
| 대메뉴 | UI 네비게이션 묶음 | 사용자가 화면을 *찾는* 방식 |
| 소메뉴 | 화면 1개 (URL 1개) + 권한 게이팅 단위 | `MenuAuthInterceptor`가 인가하는 단위 |
| 거래 | 업무 동작 1개 (조회/저장/삭제/승인 등) | Service 메서드 1개 |
| 도메인 | 쓰기로 소유하는 테이블(군) | 코드를 *묶는* 방식 |

**대메뉴/소메뉴(메뉴 축)와 도메인(데이터 축)은 다른 축이다.** 자주 비슷하지만 일치를 보장하지 않는다.
대메뉴 1개에 도메인이 여러 개일 수 있고, 도메인 1개가 여러 대메뉴에 등장할 수 있다.

## 제1원칙: 소유(own) vs 참조(reference)

화면 하나가 테이블 여러 개를 건드려도, 대개 **쓰는 건 1~2개, 나머지는 읽기(참조)** 다.
참조 테이블마다 도메인을 만들면 폭발하고, 다 한 도메인에 넣으면 god service가 된다.

| | 도메인이 된다 | 도메인 안 된다 (참조/하위) |
|---|---|---|
| 데이터 관계 | 이 기능이 **CUD(쓴다)** | **읽기만** 한다 |
| 예 | `SMS_HISTORY`를 수정 | 공통코드, 부서명(`DEP`), 사용자명(`EMP`) 조회 |
| 처리 | 자기 Service/Mapper 보유 | 기존 도메인이 조인/조회로 **빌려 쓴다** |

## 판별 절차 A — 테이블이 있을 때 (폐쇄망 본개발)

1. 화면(소메뉴)의 **거래를 전부 나열**한다.
2. 각 거래가 **무엇을 쓰는지(INSERT/UPDATE/DELETE 대상 테이블)** 만 본다. 조회 테이블은 일단 무시한다.
3. **쓰기 대상 테이블(+그 하위 테이블)별로 거래를 묶는다.** 묶음 1개 = 도메인 1개.
4. **조회 전용 거래**는, 결과로 주로 내놓는 데이터의 도메인에 붙인다.
5. 묶음이 2개 이상이면 화면 1개라도 도메인은 여러 개다. 묶음별로 Service/Mapper를 나눈다.

## 판별 절차 B — 테이블이 없을 때 (base / 설계 단계)

테이블 확정 전에는 "쓰기 주인" 기준을 쓸 수 없다. 이때는 **메뉴를 임시 프록시**로 쓴다.

1. 소메뉴(화면)의 **업무 명사**로 임시 도메인을 잡는다. (만들 테이블은 보통 업무 명사를 따라간다)
2. 임시임을 명시한다. 컬럼명/테이블명을 확정하지 않는다. (`CLAUDE.md`, `domain-rules.md` 미확정 방침)
3. 폐쇄망에서 테이블이 보이면 **절차 A로 재확정**한다.

> 도메인 경계는 base 단계에서 완벽할 필요가 없다. 제네릭 base 구조에서는
> 나중에 메서드를 Service 간에 옮기는 비용이 싸므로, 임시로 잡고 테이블 확인 후 조정한다.

## 합치기 / 나누기 체크리스트

**한 도메인으로 합쳐라 (나누지 마라):**

- 메인 테이블의 **하위 테이블** (1:N로 함께 생기고 함께 죽음 — 예: 주문↔주문상세)
- **월별/파티션** 변형 테이블 (같은 도메인. 동적 테이블명 1 VO로 처리, 테이블마다 클래스 금지)
- **읽기 전용 참조** 테이블 (공통코드, `EMP`, `DEP` 등)
- 그 화면 **하나만을 위해** 존재하는 테이블

**별도 도메인으로 나눠라:**

- **메인 테이블이 둘인데 생명주기가 다르다** (따로 생기고 따로 죽음)
- **다른 화면에서도 독립적으로 재사용**된다
- **변경 이유가 다르다** (한쪽 규칙이 바뀌어도 다른 쪽은 안 바뀜)
- 한쪽만 따로 **삭제/아카이브**하고 싶은 적이 있다

> 빠른 직관: **"이 둘을 따로 지우고 싶을 때가 있나?"** 있으면 다른 도메인, 항상 같이 죽으면 같은 도메인.

## 파일 산출 규칙 (폭발 방지)

도메인이 정해지면 파일은 다음 규칙으로만 만든다. 계층 규칙은 `project.md`를 따른다.

- **거래 1개 = Service 메서드 1개 + Mapper 메서드 1개 (+ 필요 시 쿼리 1개).** 새 클래스가 아니다.
- **도메인 1개 = Controller 1 + Service 1 + Mapper 인터페이스 1 + Mapper XML 1 + VO(결과 모양 수) + DTO(요청 모양 수).**
- **화면(소메뉴) 1개 = html 1 + js 1.** 같은 도메인의 여러 화면은 Service/Mapper를 공유한다.
- **호출처가 1곳뿐인 헬퍼**(Builder/Tree/Permission 류)는 private 메서드 또는 inner static class. top-level 파일로 빼지 않는다.
- **참조 테이블은 도메인을 새로 만들지 않는다.** 기존 도메인 Mapper에서 조인/조회로 빌려 쓰거나 공통 코드(`CommonCode`)를 쓴다.

결과적으로 파일 수는 다음 식을 따른다. **거래 수는 들어가지 않는다.**

```text
파일 수 ≈ (도메인 수 × 약 4) + (화면 수 × 약 2) + base 공통
```

## 과오 경고 (양쪽 다 금지)

- **과분해**: 참조마다·거래마다 도메인/클래스. → 폭발. 폐쇄망 개발자가 못 따라온다.
- **과결합**: 관련 없는 업무를 한 도메인에. → god service. "때려박기"의 재발.
- 스윗스팟은 항상 **"메인 테이블 1개 + 그것이 소유한 하위/참조" = 도메인 1개** (DDD 애그리거트 단위).

## 예시

"발송조회" 대메뉴, 거래 다수인 경우:

```text
[발송조회] 대메뉴
 ├ 발송이력 조회   → SMS_HISTORY 읽기   ┐
 ├ 발송 상세       → SMS_HISTORY 읽기   ├ 도메인 A: SmsHistory
 ├ 발송 통계       → SMS_HISTORY 읽기   ┘
 ├ 재발송          → SMS_REQUEST 쓰기   → 도메인 B: SmsRequest
 ├ 수신거부 관리   → BLOCK_LIST 쓰기    → 도메인 C: BlockList
 ├ 템플릿 조회     → 읽기만             → 참조 (B가 빌려 씀)
 └ 공통코드 조회   → 읽기만             → base 공통(CommonCode)
```

→ 화면(소메뉴) 여러 개 · 거래 다수 → **도메인 3개 + 참조 2개.** 거래 수가 아니라 쓰기 테이블 수가 도메인 수를 정한다.
(테이블명은 예시이며 폐쇄망 확인 전 미확정이다.)

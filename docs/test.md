스캐폴드 소스를 모두 분석했습니다. 각 항목에 대해 정확히 답하겠습니다.                                                                                                                                                                               
                                                                                                                                                                                                                                                          
     == 1. COUNT 쿼리 서브쿼리 감싸기 ==                                                                                                                                                                                                                  
                                                                                                                                                                                                                                                          
     확인 위치: MapperXmlTemplate.java:51-56                                                                                                                                                                                                              
                                                                                                                                                                                                                                                          
     생성되는 구조:                                                                                                                                                                                                                                       
                                                                                                                                                                                                                                                          
     SELECT COUNT(1) FROM (                                                                                                                                                                                                                               
         SELECT A.COL1, A.COL2, ... FROM TABLE A WHERE 1=1                                                                                                                                                                                                
             <if test="$var != null">AND ...</if>                                                                                                                                                                                                         
     ) A                                                                                                                                                                                                                                                  
     <include refid="searchConditions"/>                                                                                                                                                                                                                  
                                                                                                                                                                                                                                                          
     원인: rawQuery에 JOIN이나 서브쿼리가 있을 수 있어서, WHERE 조건이 단일 테이블이 아닐 수 있습니다. 컬럼을 안전하게 제거하려면 SQL 파싱이 필요한데, 그렇게 하면 복잡도가 크게 올라갑니다.                                                              
                                                                                                                                                                                                                                                          
     실제 영향: Oracle CBO는 SELECT COUNT(1) FROM (SELECT ...)에서 컬럼 값을 실제로 fetch하지 않습니다. 쿼리 비용 자체는 큰 차이가 없습니다. 결론: "가벼운 비효율"이지 "결함"은 아닙니다. 내가 처음에 너무 강하게 말했습니다.                             
                                                                                                                                                                                                                                                          
     == 2. searchConditions가 비어있는 문제 ==                                                                                                                                                                                                            
                                                                                                                                                                                                                                                          
     확인 결과, 현재 scaffold 템플릿에는 조건 처리 메커니즘이 두 개 있습니다:                                                                                                                                                                             
                                                                                                                                                                                                                                                          
     A) $변수 라인 → convertToDynamicSql()이 <if>로 변환 → dynamicQuery 안에 들어감 (서브쿼리 내부)                                                                                                                                                       
     B) <sql id="searchConditions"> → $변수가 없을 때만 첫 컬럼 LIKE 조건 생성 (서브쿼리 외부)                                                                                                                                                            
                                                                                                                                                                                                                                                          
     $변수가 있으면 searchConditions는 사실상 빈 <where></where>만 남습니다. 이미 A에서 처리했으니까요.                                                                                                                                                   
                                                                                                                                                                                                                                                          
     사용자 질문 "include 안에 넣어서 생성하는게 나을까?" → 네, 나을 수 있습니다. 현재 구조의 문제는:                                                                                                                                                     
     - 조건이 dynamicQuery(인라인)와 searchConditions(<sql> fragment) 두 곳에 나뉘어 있어서, 생성된 XML을 사람이 읽을 때 혼란스럽습니다                                                                                                                   
     - SmsHistoryMapper.xml(구버전 생성물)은 두 곳 모두에 조건이 중복으로 들어가 있었습니다                                                                                                                                                               
                                                                                                                                                                                                                                                          
     개선 방향: dynamicQuery에서 SELECT/FROM과 WHERE 조건을 분리하고, WHERE 조건만 searchConditions <sql> fragment로 빼면 count와 selectList가 공유할 수 있습니다. 다만 JOIN이 있는 rawQuery에서 WHERE만 안전하게 추출하는 로직이 추가로 필요합니다.      
                                                                                                                                                                                                                                                          
     == 3. ORDER BY RESULT_MSG ==                                                                                                                                                                                                                         
                                                                                                                                                                                                                                                          
     맞습니다. 사용자가 orderBy를 직접 입력한 값입니다. ScaffoldModel:81-83에서 request.getOrderBy()를 그대로 XML에 넣습니다.                                                                                                                             
                                                                                                                                                                                                                                                          
     내가 문제라고 한 이유는 단순히 "발송이력은 시간 역순이 보통인데 결과메시지 텍스트 기준 정렬은 의도적인 건지 실수인지 확인이 필요해 보였기 때문"이었습니다. 결함이 아니라 UX 확인용 코멘트였습니다. 의도적이면 문제 아닙니다.                         
                                                                                                                                                                                                                                                          
     == 4. delete 반환값 미검사 (수정 요청) ==                                                                                                                                                                                                            
                                                                                                                                                                                                                                                          
     ServiceTemplate:68-71을 보면 delete는 update와 달리 반환값을 무시합니다:                                                                                                                                                                             
                                                                                                                                                                                                                                                          
     mapper.delete(pkArgs);                                                                                                                                                                                                                               
                                                                                                                                                                                                                                                          
     update는 updated == 0 체크를 하는데 delete는 안 합니다. 스캐폴드 템플릿과 기존 SmsHistoryService 둘 다 수정하겠습니다.                                                                                                                               
                                                                                                                                                                                                                                                          
     == 5. rowNum 미사용 ==                                                                                                                                                                                                                               
                                                                                                                                                                                                                                                          
     확인 결과: VoTemplate:31이 모든 VO에 무조건 private long rowNum;을 생성합니다. 하지만 MapperXmlTemplate의 selectList에는 ROW_NUMBER() OVER(...)나 ROWNUM이 전혀 없습니다.                                                                            
                                                                                                                                                                                                                                                          
     결론: rowNum은 항상 0입니다. TUI Grid는 클라이언트에서 자체 행 번호를 표시하므로 서버 rowNum이 불필요합니다. VoTemplate에서 제거하는 것이 맞습니다.                                                                                                  
                                                                                                                                                                                                                                                          
     == 6. GlobalModelAdvice 설명 ==                                                                                                                                                                                                                      
                                                                                                                                                                                                                                                          
     상황을 구체적으로 설명하겠습니다.                                                                                                                                                                                                                    
                                                                                                                                                                                                                                                          
     GlobalModelAdvice는 @ControllerAdvice + @ModelAttribute로 동작합니다. 이 조합의 의미는: "모든 Controller의 모든 요청이 실행되기 전에, Model에 값을 채운다"입니다.                                                                                    
                                                                                                                                                                                                                                                          
     사용자가 /sms/history/data (목록조회 JSON API)를 호출할 때, 실제로는 다음이 일어납니다:                                                                                                                                                              
                                                                                                                                                                                                                                                          
     1. Spring Security 인증                                                                                                                                                                                                                              
     2. MenuAuthInterceptor.preHandle() → menuAuthService.checkAccess() → 메뉴 트리 조회 #1                                                                                                                                                               
     3. GlobalModelAdvice.addLayoutAttributes() → menuSource.getMenuTree() → 메뉴 트리 조회 #2                                                                                                                                                            
     4. GlobalModelAdvice → resolvePageAuth() → menuSource.getPermissions() → 권한 조회                                                                                                                                                                   
     5. SmsHistoryController.getData() → service.search() → 실제 비즈니스 로직                                                                                                                                                                            
                                                                                                                                                                                                                                                          
     문제는 2, 3, 4 단계입니다. JSON API(/data) 요청인데도:                                                                                                                                                                                               
     - menus 트리를 계산하지만 JSON 응답에는 전혀 사용되지 않습니다 (Thymeleaf layout용 데이터)                                                                                                                                                           
     - pageAuth도 계산하지만 JSON 응답에 사용되지 않습니다                                                                                                                                                                                                
     - 메뉴 트리 조회가 1회 (Interceptor) + 1회 (Advice) = 같은 데이터를 2회 조회합니다                                                                                                                                                                   
                                                                                                                                                                                                                                                          
     만약 MenuSource가 DB 기반이면, 매 API 호출마다 DB 쿼리가 2회 발생합니다.                                                                                                                                                                             
                                                                                                                                                                                                                                                          
     해결 방향:                                                                                                                                                                                                                                           
     - @ModelAttribute 대신 화면 Controller에만 적용되는 방식으로 변경                                                                                                                                                                                    
     - 또는 JSON 요청(Accept: application/json 또는 @ResponseBody)은 제외                                                                                                                                                                                 
     - 또는 메뉴 트리를 세션/캐시에 저장
